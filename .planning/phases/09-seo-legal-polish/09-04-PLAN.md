<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Hardcoded social URL strings → `<a href>` | Strings live in Footer.astro; if future refactor reads these from Google Sheets (like speakers/sponsors), the URLs cross an untrusted boundary |
| `safeUrl()` allowlist filter → rendered anchor | Already the closed trust boundary |
| Footer legal links → /code-of-conduct, /privacy, /terms | Internal routes; no trust boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-11 | Tampering / XSS | Social URL anchor href | mitigate | All 4 social URLs pass through `safeUrl()` which accepts ONLY `http:` or `https:` protocols. `javascript:`, `data:`, `vbscript:`, `file:` URIs → null → icon hidden. Mirror of Phase 11 SocialLinks.astro fix. |
| T-09-12 | Information Disclosure | External social links tab-napping | mitigate | Every social anchor uses `target="_blank" rel="noopener noreferrer"` — no referrer leak, no `window.opener` access from the opened tab. |
| T-09-13 | Spoofing | Legal link internal hrefs | accept | Internal routes resolved via `getLocalePath()` — framework-controlled, closed-set path strings. |
| T-09-14 | Denial of Service | Footer SVG payload size | accept | 4 inline SVGs at ~200 bytes each = ~800 bytes gzipped. Negligible. |
| T-09-15 | Repudiation | N/A | — | Footer is static; no audit trail needed. |
</threat_model>
