**Block-by-block contract**

1. **Heading h2 (centered):** Style mirrors `SponsorTierSection.astro` line 25 with `font-bold` upgrade per Typography table. Centered to set the strip apart visually from the left-aligned past-editions sections above it.
2. **Logo grid:** 4 columns desktop / 3 columns tablet / 2 columns mobile. Cards render via the same chrome as the existing `SponsorCard.astro` Platinum tier (border + bg-card + rounded-md + hover lift) but with:
   - Smaller logo bounding box (`max-w-[180px] max-h-16` vs `max-w-[220px] max-h-20` on the canonical page).
   - No sponsor description shown (homepage strip is preview-only — descriptions live on `/sponsors`).
   - No sponsor-name caption beneath the logo when the logo IS present (kept ONLY as fallback when `logoSrc === null`). This is the single visual divergence from `SponsorCard.astro` line 99 — rationale: the homepage strip is logo-led; descriptions belong on the dedicated page.
3. **CTA row:** Centered text link to `/sponsors` (or `/en/sponsors` for EN locale). Same styling pattern as Phase 23's CTA-row anchors (`text-primary hover:underline` + trailing `→` in `<span aria-hidden="true">`). NOT an internal `target="_blank"` — same-tab navigation is correct for an in-site link.

**Container width:** Outer `max-w-6xl` (matches surrounding sections). Inner logo grid `max-w-5xl` so the cards do not feel "wall-to-wall" at desktop widths — preserves breathing room around the Platinum tier (visually communicates premium-tier weight).

Per Pitfall #4 and the "Decisions to confirm" table at the bottom of this spec, the simpler path is a NEW `Edition2023Link.astro` that has exactly the props it needs (logo + logoAlt + viewPageHref + viewPageLabel + heading), rather than evolving `PastEditionMinimal.astro` to support an empty-photos variant. 2. `PastEditionMinimal` is currently consumed ONLY by the two homepage files (`grep -rn "PastEditionMinimal" src/` confirms). Phase 26 will remove both imports. Once Phase 26 ships, the file becomes orphan — Phase 26 also handles its deletion.
3. A purpose-built `Edition2023Link.astro` makes the Stitch mockup → code mapping 1:1 (no flag-driven layout branches).

`PastEditionMinimal.astro` is therefore left UNTOUCHED in Phase 24. Phase 26 imports `Edition2023Link` and removes the `PastEditionMinimal` reference; the now-orphan file is deleted in the same Phase-26 cleanup task (per Pitfall #11 — orphan-import guard).

| Key | Status | FR | EN |
|-----|--------|----|----|
| `sponsors.homepage.heading` | **NEW (D-01)** | "Nos partenaires Platinum" | "Our Platinum sponsors" |
| `sponsors.homepage.cta` | **NEW (D-02)** | "Voir tous les sponsors" *(NO trailing arrow in the value — Pattern B is final; arrow rendered as decorative `<span aria-hidden="true">→</span>` in template only)* | "View all sponsors" *(same — no trailing arrow)* |
| `sponsors.card.aria` | EXISTING (reuse) | "Visiter le site de {name} (nouvelle fenêtre)" | "Visit {name}'s website (opens in a new window)" |
| `editions.2023.compact_title` | EXISTING (reuse) | "Édition 2023 (Kubernetes Community Days France)" | "2023 Edition (Kubernetes Community Days France)" |
| `editions.2023.view_page_cta` | EXISTING (reuse) | "Voir l'édition 2023 →" | "View the 2023 edition →" |

**Arrow-glyph consistency rule (clarification for the planner):** Two patterns coexist in the codebase for trailing arrows on text-link CTAs:
- Pattern A — arrow in copy: `editions.2023.view_page_cta` ends with " →". Template MUST NOT add a second arrow span.
- Pattern B — arrow as decorative span: `editions.2026.replays_cta` does NOT end with arrow; template adds `<span aria-hidden="true">→</span>`.

Phase 24 SHOULD adopt Pattern B for the new `sponsors.homepage.cta` key (i.e. translate as plain "Voir tous les sponsors" / "View all sponsors" WITHOUT the trailing arrow, then render the arrow span in the template) — this matches Phase 23's convention and keeps the arrow accessibility-correct (aria-hidden) regardless of locale. This contradicts the "Default" copy in the table above which includes the arrow in the string. **Final planner decision (this UI-SPEC's recommendation):** translate WITHOUT trailing arrow; render arrow as decorative span in the template. The table above is updated to reflect this — see Q5 in §"Decisions to Confirm" if the user wants the alternate convention.

**i18n key drift gate (PITFALLS #2):** Both NEW keys above MUST be added to BOTH `fr` and `en` in the same commit. The planner should include a final task: "Verify `Object.keys(ui.fr).length === Object.keys(ui.en).length` after this phase ships — exactly +2 vs Phase 23 baseline."
