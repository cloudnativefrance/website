<task type="auto">
  <name>Task 2: Write CONTRIBUTING.md (rules summary + PR/commit flow)</name>
  <files>CONTRIBUTING.md</files>
  <action>
Create `CONTRIBUTING.md` at the repo root. This file is the external-contributor-facing summary of rules that otherwise live in `CLAUDE.md`. Per D-05, link to `CLAUDE.md` as the authoritative source rather than re-specifying every rule verbatim.

Structure (H1 + ~5 H2 sections, tight prose, second person, no emoji):

1. **H1: Contributing to Cloud Native Days France Website** — one-line welcome.

2. **H2: Before you start** — "Read `CLAUDE.md` — it holds the canonical Design Rules and Data Rules enforced throughout this repo. This file summarizes the same rules for humans and adds the PR/commit conventions." Link: `CLAUDE.md`.

3. **H2: The three non-negotiable rules** — each rule gets a short sub-section (bold lead-in, 2-4 sentences). Mirror CLAUDE.md wording where possible:
   - **CSVs are the single source of truth.** Never hardcode speaker/session/sponsor/team data in `.astro`, `.ts`, or `.tsx` files. Always fetch it through the CSV loader helpers. If you need to add or rename a column, update the Google Sheet, the CSV parser, the Zod schema in `src/content.config.ts`, AND every downstream consumer in the same PR. Full editor runbook: `docs/updating-content.md`.
   - **Stitch-first for any visual change.** Every new page or significant UI change is designed in Google Stitch first, validated by the user, then implemented in code. Never skip straight to code for visual work. After executing a phase that produces pages or UI components, present each page in Stitch for review before considering the work done. See `STITCH_WORKFLOW.md` for the tooling setup and `DESIGN.md` for the tokens.
   - **No hardcoded FR/EN strings.** All locale-dependent text uses i18n translation keys (see `src/i18n/`). Do not ship a string in one language only; add both FR and EN entries in the same PR.

4. **H2: PR and commit flow** — pragmatic, literal. Include:
   - Branching: work on a feature branch named for the phase/plan (e.g. `phase-14/docs`). Open a PR against `main`.
   - Commit message style: **Conventional Commits with phase-plan scope** as used throughout this repo: `docs(14-01): add root README` / `feat(04-02): speaker grid` / `fix(11-01): xss in SocialLinks` / `test(13-02): rewrite speaker tests`. One logical change per commit. Do not co-author commits.
   - Before opening a PR run: `pnpm test`, `pnpm astro check`, `pnpm build`. See `docs/testing.md` for what each covers and which pre-existing failures are known non-blockers.
   - Keep PRs focused — one phase, one plan where possible.
   - Do NOT force-push to shared branches.

5. **H2: Review flow** — "At least one maintainer review. Stitch-first changes also require the screenshot/Stitch link in the PR description so reviewers can diff design against implementation."

6. **H2: Where to go next** — short link list:
   - `README.md` — project overview and quickstart
   - `docs/repo-structure.md` — where things live
   - `docs/updating-content.md` — CSV runbook
   - `docs/testing.md` — test commands
   - `CLAUDE.md` — canonical rules (authoritative)

Tone: pragmatic, second-person. No emoji. No "Welcome to the project!!!" fluff. Do NOT contain `TODO`, `FIXME`, or `TBD`. Do NOT re-specify design-system colors or tokens — those live in `DESIGN.md` (D-05).
