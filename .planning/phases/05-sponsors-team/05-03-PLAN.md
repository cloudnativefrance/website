<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CSV social_* columns → SocialLinks anchor href | Admin-supplied URLs surface as clickable links |
| CSV photo column → SpeakerAvatar img src | Admin-supplied path/URL surfaces as image |
| CSV role_fr/en, name → DOM text | Admin-supplied text surfaces to user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Injection / XSS | SocialLinks anchor href (team member socials) | mitigate | SocialLinks.astro already enforces safeUrl() allowlist on every href (Phase 11 work). Team schema Zod additionally enforces `.url()` parsing at build time via `socialUrl`. Double-layered — no new work needed. |
| T-05-09 | Injection | SpeakerAvatar photo src | mitigate | Photo paths in the team CSV are admin-controlled. SpeakerAvatar renders a plain `<img>` with the path; since Zod schema declares `photo` as `z.string().optional()` with no URL filtering, a malicious path like `javascript:` cannot reach an `<img src>` attribute (browsers ignore `javascript:` in img src), but `data:` URLs could. This plan adds a build-time doc comment in team.csv / content.config.ts recommending paths stay under `/team/*.jpg` and never schemes other than `/` or `https:`. Runtime attack surface is limited since the image does not execute script; disposition: accept residual risk for v1.0, revisit if exposure grows. |
| T-05-06 | XSS | name, role text → DOM | accept | Astro auto-escapes. No `set:html` used. |
| T-05-10 | Repudiation / UX | Card hover lift on non-clickable card | mitigate | UI-SPEC explicitly forbids hover lift on team cards (dark pattern — affords clickability that doesn't exist). This plan enforces: TeamMemberCard has no `hover:-translate-y` class. |
</threat_model>
